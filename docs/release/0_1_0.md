# Release Notes — Mandaí v0.1.0

**Data:** 2026-05-11
**Escopo:** MVP educacional — aplicativo web de pedidos pickup-only, stack DDD/Clean Architecture enxuta para demo de mentoria.

---

## Funcionalidades implementadas

### Backend — `apps/api` (Fastify 5 + TypeScript + Prisma 5 + Neon Postgres)

**Infraestrutura**
- Scaffold completo do projeto: `package.json`, `tsconfig.json` (strict, CommonJS, ES2022), `.env.example`, `vercel.json` (catch-all rewrite com warm-start Fastify singleton)
- `README.md` com instruções completas de setup local, Neon, seed e deploy Vercel

**Banco de dados**
- `prisma/schema.prisma` — 5 modelos: `Restaurant`, `MenuSection`, `MenuItem`, `Order`, `OrderItem`; alinhado ao `docs/erd.md` incluindo campos extras (`address`, `neighborhood`, `phone`, `customerName`, `subtotalCents`, `discountCents`, `couponCode`, `qrPayload`, `estimatedReadyAt`, `OrderItem.note`, `modifiers` JSON)
- `prisma/seed.ts` — 8 restaurantes cobrindo todas as categorias do handoff (pizza, hambúrguer, japonesa, mexicana, padaria, brasileira, açaíteria, saudável), ~40 itens de cardápio; 1 restaurante propositalmente fechado (El Taco Loco) para exercitar o estado de restaurante fechado

**Arquitetura DDD/Clean (bounded context `ordering`)**
- `domain/entities`: `Restaurant`, `MenuItem`, `Order` (+ tipos `OrderItem`, `OrderModifier`, `OrderStatus`)
- `domain/value-objects`: `Money` (valida inteiro não-negativo, formata BRL, `add`/`subtract`)
- `domain/repositories`: `IRestaurantRepository` (`findAll`, `findById`, `search`) e `IOrderRepository` (`create`, `findById`) — interfaces puras, zero imports de Fastify/Prisma
- `application/use-cases`: `ListRestaurantsUseCase`, `GetRestaurantUseCase`, `SearchUseCase`, `CreateOrderUseCase`, `GetOrderUseCase` — cada um recebe dependências no construtor, lança `HttpError` diretamente
- `infra`: `PrismaRestaurantRepository`, `PrismaOrderRepository` — `toDomain()` inline no fim de cada arquivo, sem classe `Mapper` separada
- `http/ordering.routes.ts` — plugin Fastify com schemas Zod validando params/query/body/response
- `ordering.module.ts` — `buildOrderingModule(prisma)`: DI manual e explícita, sem decorators

**Endpoints (todos sob `/api`)**

| Método | Path | Cobertura |
|---|---|---|
| GET | `/api/restaurants` | Lista com filtro `?category=` (US-01, US-02) |
| GET | `/api/restaurants/:id` | Detalhe + cardápio completo com seções e itens (US-04) |
| GET | `/api/search?q=` | Busca case-insensitive em nomes de restaurante e prato (US-03) |
| POST | `/api/orders` | Cria pedido, valida restaurante aberto e itens disponíveis, gera código `MA-XXXX`, retorna `qrPayload` + `estimatedReadyAt` (US-08) |
| GET | `/api/orders/:id` | Consulta pedido por ID (US-08) |
| GET | `/health` | Health check operacional |

**Gerador de código pickup**
- Charset `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (sem `0`/`O`, sem `1`/`I`), 4 chars → formato `MA-XXXX`
- 32⁴ = 1.048.576 combinações possíveis; suficiente para volumes de MVP

### Frontend — `apps/web` (Next.js 15 App Router + TypeScript)

**Infraestrutura**
- Scaffold com App Router, sem Tailwind, ESLint configurado
- `src/styles/tokens.css` e `src/styles/app.css` copiados verbatim do handoff
- `next.config.ts` com `remotePatterns` para `images.unsplash.com`
- `src/app/providers.tsx`: `QueryClientProvider` + `CartProvider`

**Rotas implementadas**

| Rota | Tipo | Cobertura |
|---|---|---|
| `/` | Server Component | Home: hero, 8 tiles de categoria, grid de restaurantes (US-01) |
| `/categoria/[slug]` | Server Component | Grid filtrado por categoria, empty state (US-02) |
| `/busca` | Server Component | Resultados com highlight `<mark>` em matches, empty state (US-03) |
| `/restaurante/[id]` | Server + Client | Layout 3 colunas: nav sticky de seções, itens, MiniCart (US-04, US-05) |
| `/sacola` | Client Component | Itens com steppers, campo de nome, resumo de pedido, "Finalizar" (US-06) |
| `/confirmacao/[orderId]` | Server Component + `ConfirmationClient` | Código MA-XXXX em JetBrains Mono com botão de cópia, QR real via `qrcode.react` + `qrPayload` do backend, "Pronto ~18 min" calculado de `estimatedReadyAt`, lista de itens com snapshots de nome e preço, total em BRL (US-08) |

**Sistema de sacola (CartContext)**
- `useReducer` + sync `localStorage` — sem Zustand
- Ações: `ADD_ITEM`, `REMOVE_ITEM`, `UPDATE_QTY`, `CLEAR_CART`
- Regra mono-restaurante: `wouldSwitchRestaurant()` helper + `ClearCartModal` para confirmação (US-10)
- Empty state na `/sacola` vazia: ilustração + CTA "Bora escolher um rango?"

**Componentes interativos (Client Components)**
- `AddItemModal`: overlay com foto, descrição, grupos de modificadores (radio/checkbox), observações (max 140 chars), stepper de quantidade (1–20), total em tempo real
- `MiniCart`: sidebar sticky na página do restaurante mostrando itens do contexto, subtotal e link para `/sacola`
- `ClearCartModal`: confirmação de troca de restaurante, fecha com Esc ou backdrop click

**Estados de erro (US-09)**
- `src/app/error.tsx`: boundary global com mensagem "Ih, deu ruim aqui." + digest code + retry + link home
- `error.tsx` por rota em `/restaurante/[id]` e `/busca`
- `not-found.tsx` para restaurante desconhecido e pedido desconhecido
- Badge "Fechado agora" + banner informativo em restaurante fechado; botões de adicionar ocultados
- Badge "Esgotado" em itens indisponíveis; botão de adicionar desabilitado

**Build e lint**
- `npm run build` → zero erros
- `npm run lint` → zero warnings

## Dúvidas técnicas surgidas durante a implementação

| ID | Dúvida | Resolução |
|---|---|---|
| TEC-001 | **Fastify 4 vs 5**: `fastify-type-provider-zod@4` exige Fastify ≥ 5.0.0 como peer. | Adotado Fastify 5 + `@fastify/cors` 10. Sem breaking changes na API pública. Documentado em **ADR-0008** (`docs/adr/0008-fastify-5-type-provider-zod.md`). |
| TEC-002 | **SQLite como fallback para dev local**: possível? | Não implementado — `prisma` usa `mode: 'insensitive'` na busca, que é PostgreSQL-only. Neon free tier é obrigatório e suficiente; documentado no `README.md`. Alinhado com ADR-0004. |
| TEC-003 | **Unicidade do código `MA-XXXX`**: garantia de colisão? | `Math.random()` com charset de 32 chars → 32⁴ combinações. Sem retry loop formal: colisão lançaria `unique constraint` genérico do Prisma (não tratado especificamente). Probabilidade desprezível no MVP. |
| TEC-004 | **`discountCents` aceito do caller vs travado em 0**: onde vive a regra? | Regra de negócio vive no backend: `discountCents = 0` sempre em `CreateOrderUseCase`. Campo removido do schema Zod do `POST /api/orders` — qualquer valor enviado pelo caller é rejeitado na validação. Ensina o padrão correto de DDD: regra de desconto não pode ser delegada ao cliente. |
| TEC-005 | **Scrollspy na nav de seções do restaurante**: `useIntersectionObserver` no MVP? | Não implementado para preservar o budget de Server Components. A nav destaca sempre a primeira seção; interação de scroll não atualiza o item ativo. Candidato a v0.2. |
| TEC-006 | **Modifier groups no `AddItemModal`**: backend retorna grupos de modificadores? | `GET /api/restaurants/:id` não retorna `modifierGroups` nos itens (não está no schema Prisma). O modal renderiza corretamente sem grupos (foto + descrição + observação + qty). Tipo `MenuItem.modifierGroups?` deixado opcional para extensão futura. |
| TEC-007 | **CORS em produção**: como configurar? | `@fastify/cors` lê `process.env.CORS_ORIGIN`; em dev usa `"*"`. Se `CORS_ORIGIN === "*"` e `NODE_ENV === "production"`, o servidor emite `app.log.warn` alertando o operador. Documentado em `.env.example` com nota "REQUIRED in production". **Pré-requisito de deploy**: configurar `CORS_ORIGIN` com a URL do `apps/web` na Vercel. |

## Versão executiva (stakeholders)

O Mandaí v0.1.0 entrega o fluxo completo de pedido pickup-only via web, do zero ao código de retirada `MA-XXXX`:

**O que funciona de ponta a ponta:**
1. Cliente acessa a Home, vê restaurantes próximos por categoria
2. Filtra por tipo de cozinha ou busca por prato/restaurante
3. Entra no cardápio, customiza item (acompanhamentos, observações, quantidade) e adiciona à sacola
4. Sacola respeita a regra de um restaurante por vez (avisa antes de trocar)
5. Confirma o pedido com o nome, recebe um código `MA-XXXX` e QR code para apresentar no balcão
6. Estados de erro cobertos: restaurante fechado, item esgotado, busca sem resultado, erros técnicos

**O que não entra nesta versão** (intencional):
- Autenticação/login
- Pagamento online (pickup paga no balcão)
- Cupons de desconto
- Responsividade mobile (desktop 1440px)
- Gestão de bairro/endereço de entrega

**Stack:** Next.js 15 (frontend Vercel) + Fastify 5 (API Vercel) + Neon Postgres — dois projetos independentes no mesmo repositório, prontos para deploy com um clique.

## Riscos mapeados

| Severidade | Risco | Mitigação sugerida |
|---|---|---|
| 🟡 Médio | **Colisão de código `MA-XXXX`**: sem retry loop formal; colisão lança exceção genérica do Prisma | Adicionar try/catch específico para `P2002` (unique constraint) em `CreateOrderUseCase` com até 3 tentativas de geração; v0.2 |
| 🟡 Médio | **CORS `"*"` em produção**: se `CORS_ORIGIN` não for configurado na Vercel antes do deploy, a API aceita chamadas de qualquer origem. O servidor agora emite `log.warn` em produção se a variável não estiver configurada. | Configurar `CORS_ORIGIN` como variável de ambiente obrigatória no painel da Vercel antes do go-live |
| 🟡 Médio | **Sem rate limiting em `POST /api/orders`**: endpoint de criação de pedido não tem proteção contra abuso | Adicionar `@fastify/rate-limit` pós-MVP (ex: 10 pedidos/minuto por IP) |
| 🟡 Médio | **Imagens Unsplash hardcoded**: IDs de imagem retirados do handoff como placeholder | Substituir por fotos licenciadas ou banco de imagens próprio antes do go-live público |
| 🟢 Baixo | **Scrollspy não funcional**: nav de seções no restaurante destaca sempre a primeira seção | Implementar `useIntersectionObserver` em v0.2 |
| 🟢 Baixo | **Modifier groups ausentes no schema**: `AddItemModal` renderiza sem grupos de modificadores | Adicionar `ModifierGroup` e `Modifier` ao schema Prisma + endpoint em v0.2 |
| 🟢 Baixo | **Neon obrigatório para dev local**: sem fallback SQLite | Documentado; Neon free tier basta. Alternativa: Postgres local via Docker |
| 🟢 Baixo | **Logo no header via `next/image`**: SVG renderizado como `<img>` pode ter comportamento diferente de inline SVG | Monitorar em produção; trocar por inline SVG se houver problema de rendering |
| 🟢 Baixo | **`useRestaurants` / `useRestaurant` hooks dead code**: definidos mas nunca chamados — pages usam `fetch()` server-side (comportamento correto) | Remover ou documentar como utilitários para uso client-side futuro em v0.2 |
| 🟢 Baixo | **`Restaurant.ensureOpen()` não chamada no use case**: `CreateOrderUseCase` verifica `!restaurant.isOpen` diretamente em vez de usar o método da entidade | Uniformizar: chamar `restaurant.ensureOpen()` no use case para reforçar o padrão de lógica no domínio |
| 🟢 Baixo | **`qrPayload` com domínio hardcoded** (`mandai.app`): sem env var configurável | Extrair para `process.env.APP_DOMAIN` antes do go-live público |
